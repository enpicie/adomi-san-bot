# General Platform Spec

*Stack-agnostic. Applies to every project regardless of language, framework, or cloud provider.*
*Pair with a stack spec and an app spec. Feed all to Claude Code on init.*

---

## 1. Purpose

This spec encodes platform engineering standards that apply universally. It defines the philosophy. Stack specs implement it. App specs define what gets built.

It is one of four general specs that compose into every project init:

- **General Platform Spec** (this file) — project scaffolding, docs, environments, gitignore, observability
- **General Coding Standard** — naming, SOLID, DRY, single source of truth, error handling, secrets in code, documentation generation, testing
- **General API Standard** — route naming, route registry as the source of API availability, request/response contract, auth/access, DB connection management
- **General Design Standard** — layout modes, adaptive layout, tokens, interactive states, cross-screen consistency
- **General Mobile UX Standard** — touch/pointer dual input contract, native gesture mechanics (swipe-reveal, long-press drag, pinch), touch targets, keyboard/viewport behavior — included whenever touch usage is expected

Claude Code must follow every rule in this spec unless a stack spec explicitly overrides it with a documented reason.

## 2. On Init — What Claude Code Must Always Generate

When initializing any project from this spec, Claude Code must generate the following before writing any application code:

### 2a. CLAUDE.md (committed)

A `CLAUDE.md` file at the project root. This is Claude Code's persistent memory for this project. It must include:

- Project name and one-line description
- Stack overview (what and why)
- Directory structure summary
- Key conventions and patterns used in this codebase
- Security rules (non-negotiable, listed explicitly)
- What Claude Code must NOT modify without explicit instruction
- Pointers to `.platform/` docs for deeper context
- Reference to which specs were used to initialize this project

`CLAUDE.md` is committed to version control. Every developer and every Claude Code session shares the same project context automatically.

### 2b. .claude/ (always gitignored)

`.claude/` contains Claude Code session memory, local customizations, and conversation history. It is machine-specific and may contain sensitive context. Never committed. Always in `.gitignore`.

### 2c. docs/ (committed) — Learning Documentation

A `docs/` folder at the project root. These are learning documents — they exist to orient developers to the tools, patterns, and decisions in the project. They are not API references. They are not wikis. They explain *why* things are the way they are, with links to go deeper.

**Every file in `docs/` must begin with this exact disclaimer:**

```
> AI-GENERATED ORIENTATION DOCUMENT
> This document was generated by Claude Code as a learning aid during project initialization.
> It is intended to orient developers to the tools and patterns used in this project.
> It reflects the project as understood at initialization time and may not capture later changes.
> Always verify against official documentation before making decisions.
> Official documentation links are provided throughout this document.
```

**Required docs on init.** Stack spec adds stack-specific files on top of these.

`docs/architecture.md`

- How the system fits together — what each piece is and why it exists
- Data flow: how a request moves from the user through the system and back
- Key technical decisions and the rationale behind them (links to ADRs in `docs/decisions/`)
- Links to official documentation for each major component

`docs/contributing.md`

- Local setup — assumes only the language runtime and git are installed
- Branch conventions: `feature/`, `fix/`, `chore/`
- PR process: what CI checks run, what is required before merging
- How to add a new feature following the patterns already in use
- How to add a new database table following the project's migration and RLS conventions
- Common mistakes and how to avoid them

**Quality bar for learning docs:**

- Every tool and pattern introduced must include a link to official documentation
- Explanations assume the reader is unfamiliar with the specific tool — not with software in general
- No doc should leave a reader wondering where to go to learn more
- If a pattern is project-specific (not in official docs), explain it fully inline

### 2d. .platform/ (gitignored — sensitive architectural docs)

Architectural documentation that may contain sensitive decisions, internal data models, or cost economics. **Gitignored by default.** Not committed unless the project is open source and the owner explicitly removes it from `.gitignore` with a documented reason.

These files are for the owner and for Claude Code — not for public consumption.

Required files on init:

`.platform/data-model.md`

- Complete entity list with all field names, types, constraints, and defaults
- Relationships between entities — Mermaid diagram preferred
- Notes on soft deletes, audit fields, ownership patterns
- Updated whenever the schema changes — this is the source of truth for the data model

`.platform/scaling-plan.md`

- Current architecture diagram
- Scaling thresholds: what metric triggers moving to the next tier
- Cost estimate per tier
- Concrete action to take at each tier (upgrade, add service, rearchitect)
- Named owner responsible for initiating scaling review
- Decision on staging environment: when it gets added and what triggers that decision

### 2e. README.md

Required sections in order:

1. Project name + one-line description
2. Status badge: `Alpha` | `Beta` | `Production`
3. Local Setup — assumes only the language runtime and git
4. Environment Variables — link to `.env.example`, explain non-obvious keys
5. Running the App — single command, what URLs open
6. Running Tests
7. Deployment — how and where
8. Initial Setup TODOs — populated on init, removed as completed
9. Architecture Notes — brief, links to `docs/` and `.platform/`

### 2f. Makefile

Machine-agnostic local development. These four targets must always exist with exactly these names:

```makefile
dev:      # start full local stack
test:     # run test suite
lint:     # run linter
build:    # production build
```

Stack spec defines what each target actually runs.

### 2g. .env.example

Committed. Contains every environment variable the project needs with empty values and inline comments explaining each one. No real values ever committed. Stack spec defines the variables.

### 2h. .github/deployment.env

Committed. Contains project-level CI/CD configuration sourced by the `config` job in `release.yml`. No secrets — only non-sensitive project identifiers.

```bash
APP_NAME=         # project identifier, used in state keys and resource names
AWS_REGION=       # primary deployment region
DEPLOYMENT_ENV=   # prod (staging added when scale justifies it)
```

### 2i. ADRs — Architecture Decision Records

Every tool choice made by the spec has a corresponding ADR generated on init. Lives in `docs/decisions/`. Format:

```markdown
# ADR-NNN: [Decision Title]

## Status
Accepted

## Date
YYYY-MM-DD

## Context
Why this decision was needed.

## Decision
What was decided.

## Rationale
Why this option over alternatives.

## Consequences
Tradeoffs accepted.

## Supersedes / Superseded By
```

ADRs are committed. They are the audit trail for architectural choices. When choices change, the old ADR is marked `Superseded By: ADR-NNN` and a new one is created.

## 3. CLAUDE.md Requirements (detail)

The generated `CLAUDE.md` must contain at minimum:

```markdown
## Project
[Name] — [one-line description]
Initialized with: spec-general.md + [stack spec] + [app spec]

## Stack
[Stack summary — what each piece is and why]

## Structure
[Directory tree, annotated with purpose of each directory]

## Security Rules — Non-Negotiable
[From stack spec — listed explicitly here too, not just referenced]

## Conventions
[Key patterns: how to add a route, how to add a table, naming conventions, etc.]

## Design Standards
Source: scroll-design-general v[version]

### Project Color Palette
[All semantic token names and values — derived from config or confirmed with user]

### Spacing Scale
4px base unit: 4, 8, 12, 16, 24, 32, 48, 64, 96. No arbitrary values.

### Typography
[Typefaces, size scale, weight usage]

### Border Radius Scale
[Radius values in use]

### Z-Index Scale
base | dropdown | sticky | modal | toast — [values]

### Project-Specific Overrides
[Any intentional exceptions to the design standard, with rationale]

### Enforcement Rules (Always Active)
- All inline elements (text next to icon/badge/tag/avatar) are vertically centered
- All interactive states implemented: hover, focus, active, disabled, loading
- No hardcoded color values in component files — tokens only
- No outline:none without a custom focus replacement
- No empty states left as blank containers
- No horizontal scroll at any standard viewport width
- Content max-width enforced — no edge-to-edge stretch on wide screens
- No !important, no ID selectors, no selectors deeper than 3 levels
- transition: all is never used — always specify the property
- Font sizes in rem only
- Every flex container has explicit align-items set

## Do Not Modify Without Explicit Instruction
[List files/dirs Claude Code should never touch autonomously]

## Deeper Context
- .platform/data-model.md — full entity list, relationships, schema
- .platform/scaling-plan.md — cost tiers, thresholds, scaling actions
- docs/ — architecture, contributing, stack-specific learning guides
- docs/decisions/ — ADRs for every architectural choice
```

**Self-installation rule for design standards:** When any spec paired with this general spec is first used in a project, Claude Code must check whether `CLAUDE.md` has a `## Design Standards` section. If not, add it with project-specific values filled in by inspecting the codebase config or asking the user. If it exists, review and update. Never silently deviate from it — all intentional overrides are documented in that section with rationale before or immediately after implementation.

## 4. Observability — Required from Day One

**Structured logging.** A logger utility is created at init. All log entries are structured JSON. Every meaningful operation logs at minimum: event name, user ID if applicable, timestamp, environment. Stack spec defines implementation.

**Request monitoring and error tracking.** Handled by infrastructure-level tooling — never built manually. Stack spec defines the specific tools for the stack.

---

## 5. Environments

Two environments minimum: `local` and `production`. No staging at MVP — it doubles infrastructure cost for personal projects. Document this decision explicitly in `.platform/scaling-plan.md` with criteria for when staging gets added.

---

## 6. CI/CD

All projects deploy using the org platform composite actions and Terraform conventions defined in `spec-platform-cicd.md`. That spec is the single source of truth for CI/CD patterns — workflow structure, Terraform conventions, IAM roles, state keys, and composite action usage.

`spec-platform-cicd.md` must be included alongside this spec for every project init.

## 9. Anti-Patterns — Never Do These

These are explicitly prohibited. No instruction, no framing, and no "it's simpler for now" justification overrides them.

- **Never implement observability via database tables.** No `usage_events`, no `request_logs`, no `error_logs` in the DB. Monitoring tools exist for this. Use them.
- **Never write "follow best practices" as an instruction.** It is meaningless. Every decision must be explicit — name the tool, name the pattern, name the reason.
- **Never hand-roll auth.** Auth is handled by the auth provider defined in the stack spec. No custom session logic, no custom token handling.
- **Never use `console.log` directly in application code.** Always use the structured logger utility.
- **Never hardcode secrets or API keys.** All secrets live in environment variables. Never in code, never in comments.
- **Never skip RLS.** Every table has RLS policies in the same migration. No exceptions.

---

## 10. Decided Against

Explicit record of patterns considered and rejected. Claude Code must not reintroduce these.

| Pattern | Rejected Because |
|---|---|
| `usage_events` DB table for monitoring | Monitoring tools (CloudWatch, Datadog, Vercel Analytics) are purpose-built for this. A DB table is slower, harder to query, adds schema overhead, and provides worse insight than any real monitoring tool. |
| "Follow best practices" instructions | Meaningless. Claude Code must be given explicit decisions, not delegated judgment calls. |
| Staging environment at MVP | Doubles infrastructure cost for personal projects. Local + prod only until scale justifies it. |

---

## 11. Audit Mode

When applied to an existing repository, Claude Code must:

1. Read this spec and all paired specs in full
2. Scan the repository structure
3. Produce `.platform/audit.md` with:
   - Standards already met
   - Partially met — specific gaps noted
   - Not met — recommended remediation
4. Ask which gaps to address automatically vs flag for manual review
5. Apply approved changes in a single commit: `chore: platform conformance updates`
